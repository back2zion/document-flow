FAQ
파일 처리 제한 및 용량
사용자가 업로드할 수 있는 파일 크기와 개수 제한은 어떻게 되나요?

사용자 업로드는 파일당 100MB 이하, 최대 5개 파일까지 가능합니다. 이는 서버 리소스를 효율적으로 관리하고 처리 시간을 최적화하기 위한 제한입니다. 관리자 업로드의 경우 별도의 배치 처리 시스템을 통해 더 큰 용량과 많은 수의 파일을 처리할 수 있습니다.

대용량 파일 처리 시 메모리 문제는 어떻게 해결하나요?

대용량 파일 처리 시 다음과 같은 메모리 관리 전략을 사용합니다:

  • 스트림 처리: 파일 전체를 메모리에 로드하지 않고 청크 단위로 스트림 처리
  • 메모리 한계 설정: 각 처리 작업에 메모리 상한선 설정
  • 가비지 컬렉션 최적화: 주기적인 가비지 컬렉션으로 미사용 메모리 해제
  • 임시 파일 사용: 필요 시 디스크 기반 임시 파일 활용

이러한 전략을 통해 OOM(Out of Memory) 오류를 방지하고 안정적인 파일 처리를 보장합니다.

성능 및 확장성
한 번에 몇 개의 파일까지 동시 처리 가능한가요?

시스템은 서버 리소스에 따라 동적으로 조정되지만, 기본 설정은 다음과 같습니다:

  • 사용자 요청: 동시에 최대 10개 요청(각 최대 5개 파일) 처리
  • 관리자 배치 처리: 서버 부하에 따라 20-50개 파일 동시 처리
  • 자동 조정: 서버 부하에 따라 처리량 자동 조절

처리 큐를 사용하여 요청이 많을 경우 대기열에 추가되며, 시스템 부하에 따라 처리 속도가 조절됩니다.

파일 형식이나 HWP 버전 업데이트에 어떻게 대응하나요?

파일 형식 및 버전 변경에 대응하기 위한 전략:

  • 모듈식 파서 설계: 파일 형식별 파서를 모듈화하여 개별 업데이트 가능
  • 버전 감지 로직: 파일 헤더 분석을 통한 버전 자동 감지
  • 폴백 메커니즘: 새 버전 처리 실패 시 대체 파서 사용
  • 정기 업데이트: 한컴 공식 문서 기반 파서 정기 업데이트

특히 HWP의 경우, 5.0 이전/이후 버전과 HWPX(XML 기반) 형식에 대한 별도 처리 로직을 갖추고 있습니다.

오류 처리 및 보안
손상된 파일이나 비정상 형식은 어떻게 처리하나요?

손상된 파일 처리를 위한 안전장치:

  • 파일 무결성 검사: 파일 구조 검증으로 손상 여부 확인
  • 예외 처리: 파싱 오류 발생 시 세부 정보 로깅 후 안전하게 처리 중단
  • 부분 추출: 손상된 파일에서도 읽을 수 있는 부분 최대한 추출
  • 사용자 피드백: 파일 손상 시 명확한 오류 메시지 제공

비정상 파일 처리 시 시스템 전체에 영향이 없도록 격리된 환경에서 처리합니다.

메모리 내 처리 중 민감 정보 보호는 어떻게 하나요?

메모리 내 데이터 보호 방법:

  • 메모리 암호화: 민감한 데이터는 메모리 내에서도 암호화
  • 메모리 초기화: 사용 후 메모리 영역 명시적 초기화(zeroing)
  • 접근 제한: 메모리 덤프 및 디버깅 도구 접근 제한
  • 컨테이너 격리: 각 처리 작업은 격리된 컨테이너에서 실행

특히 사용자 업로드 파일은 처리 완료 후 메모리에서 즉시 제거되며, 관련 모든 메모리 공간이 초기화됩니다.

AI 및 기술 구현
이 시스템에서 'AI'의 역할은 정확히 무엇인가요?

이 시스템에서 AI는 다음과 같은 역할을 담당합니다:

  • 텍스트 임베딩 생성: 문서 내용을 의미적 벡터로 변환(NeoAli RAG)
  • 유사도 분석: 문서 간 의미적 유사성 판단(중복 검사)
  • 개인정보 탐지: 텍스트 내 개인정보 패턴 인식 및 분류
  • 문서 분류: 내용 기반 문서 자동 분류 및 태깅
  • 질의응답: 문서 기반 질문에 대한 정확한 답변 생성

단순한 파일 파싱을 넘어, 자연어 처리와 기계학습 기술이 문서 내용의 의미적 이해와 활용에 핵심적 역할을 합니다.

MinIO와 같은 저장소 접근은 어떻게 관리되나요?

저장소 접근 관리 방식:

  • 역할 기반 접근 제어(RBAC): 사용자 역할에 따른 권한 설정
  • 임시 자격 증명: 필요한 작업에만 유효한 단기 접근 토큰 발급
  • 암호화 전송: TLS/SSL을 통한 모든 데이터 전송 암호화
  • 접근 감사: 모든 저장소 접근 로깅 및 모니터링
  • 네트워크 격리: MinIO 서버는 내부 네트워크에서만 접근 가능

특히 사용자 업로드 파일은 분리된 임시 버킷에 저장되며, 해당 버킷은 정기적으로 자동 정리됩니다.
